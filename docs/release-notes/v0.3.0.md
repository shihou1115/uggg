## 概要

v0.2.0 (2026-07-18) に続く機能リリース。**「朝・夜の定例会話」と「天気・生活情報」**
（spec §4.7、設計正本 docs/regular-talk-design.md v1.1）を追加した。日常支援 Tier S
（v0.2.0）で用意した予定・ToDo・リマインダー・カレンダーに **天気** という新しい材料を加え、
**1 日の始まりと終わりにキャラクターがその日の情報をまとめて話しかける**同居感の完成形を
実装する第 2 弾。

すべての基盤は low（ローカル・決定論）で動作し、AI 非依存（spec §4.2.1 不変条件）。
天気はキー不要・無料の公開 API（Open-Meteo）を使い、地域の手動設定（＝同意）まで既定オフ。
定例会話・降雨の一言はいずれも既定オフのオプトインで、新設の発話カテゴリとして既存の
発話ガバナンス（v0.2.0）にそのまま従う。

DB スキーマは v8 のまま（**新テーブルなし・マイグレーションなし**）。設定は serde default で
自動移行される。

## 変更点

### 天気・降雨の一言 (M11)

- **Open-Meteo**（キー不要・無料の公開気象 API）から今日・明日の予報を取得。地名検索から
  地域を登録する UI を持つ。**地域は手動設定・既定オフ**で、地域を選んだ時点で同意とみなす。
  外部へ送るのは天気取得のための地名・座標のみで、**座標は市区町村粒度（小数 1 桁 ≒ 約 11km）
  に丸めて**送信する（丸め前の値は保持しない）
- **降雨の一言**（既定オフ）: その日に降雨が見込まれる朝に一度だけ知らせる状況発話
  カテゴリ。カレンダーに当日の時刻付き予定があれば言い回しを強める（`weather_rain_outing`）
- 予報は定期取得（3 時間間隔）＋ 定例会話の直前に更新。オフライン時はキャッシュを使い、
  **取得から 6 時間を超えた古いキャッシュは使わず天気項目を省く**
- **出典表示**: 天気が有効な間、画面下端に「天気: Open-Meteo.com」を常時表示（CC BY 4.0 の
  帰属義務。VOICEVOX クレジットと同型）。設定パネルに正式表記とライセンスリンクを併記
- **advanced 上乗せ**: 「今日の天気は？」への応答にキャッシュを会話材料として注入

### 朝・夜の定例会話 (M12)

- **朝と夜の 1 日 2 枠**、それぞれ独立に有効/無効・時刻（既定 朝 8:00 / 夜 21:00）・
  話す曜日を設定できる。**既定は両方オフのオプトイン**
- **配達タイミング**: 「設定時刻きっかり」ではなく、設定時刻を過ぎて最初に PC を
  使っているときに 1 回配達（朝 8:00 に PC が点いていない日常を許容）。**設定時刻から
  6 時間で当日分は失効**（夜遅くの「おはよう」・朝夜の連続配達を防ぐ）。1 枠 1 日 1 回
- **中身（low = 定型フォーマット・ローカル完結）**: 朝 = あいさつ ＋ 今日の予定 ＋
  今日の ToDo 件数 ＋ 未完了リマインダー（直近数件に絞る）＋ 今日の天気。夜 = ねぎらい ＋
  今日の完了実績 ＋ 持ち越し ＋ 明日の予定 ＋ 明日の天気。材料が空の項目は黙って省き、
  全部空でも短いあいさつは成立する
- **二重告知の吸収**: 朝の定例会話が有効な間は、朝の ToDo 件数告知（v0.2.0）と降雨の一言を
  定例会話にまとめ、単独発火させない
- **🔕 フィードバック**: 各枠を対象に、3 回で当該枠を自動オフ（1 日 1〜2 回の定例のため、
  状況発話の間隔バックオフは適用しない）
- **advanced 上乗せ**: 同じ材料を会話調の言い回しに整形（降格中・API エラー・タイムアウト・
  空応答では low の定型文へフォールバック。**材料の選定と発話可否は常に low が確定**）

### UI・設定

- 設定「日常支援」ページに **「朝・夜の定例会話」節**（朝/夜の有効・時刻・曜日トグル・
  夜間静音との重なり警告）と **「天気（生活情報）」節**（地名検索・候補選択・いま取得・
  解除・降雨の一言・出典表記）を追加
- ステージ下端に天気クレジット `天気: Open-Meteo.com` を追加（天気有効時のみ表示）
- 取扱説明書（同梱 manual.md）を v0.3 の機能に合わせて更新

## 契約の変更（後方互換）

- **Tauri コマンド 2 件追加**: `search_location(query)`（地名 → 候補）/ `get_weather()`
  （現在の予報取得。設定 UI の検証・現在値表示用）
- **イベント追加なし**（既存 `settings-changed` のみ使用）
- **Settings に 11 フィールド追加**（`regular_morning_enabled` / `_time` / `_days`、
  `regular_evening_enabled` / `_time` / `_days`、`weather_enabled`、`weather_latitude`、
  `weather_longitude`、`weather_place_name`、`situation_rain_enabled`。既存設定は
  serde default で自動移行）
- **発話カテゴリ 9 → 12**（`SituationRain` / `RegularMorning` / `RegularEvening` を追加）。
  🔕 フィードバックの適用対象を `is_situation()` から `feedback_target()` へ分離し、定例会話
  2 枠も 🔕（3 回で該当枠 OFF）の対象に含めた。間隔バックオフは従来どおり Situation* のみ
- **辞書 events キー 4 種追加**（`regular_morning` / `regular_evening` / `weather_rain` /
  `weather_rain_outing`）。すべて省略可能で、無い辞書では無言にフォールバック（既存ゴーストは
  従来どおり動作）
- **DB スキーマ変更なし（v8 維持・新テーブルなし）**。天気キャッシュは app_settings の
  単一キー JSON（`weather_cache`）に保持し、per-day 管理も既存の app_settings キー方式に乗せる
- **静音プリフィルタ `ambient_quiet_now` を新設**（v0.3 リリース監査で追加）: 定例会話・
  降雨の一言の材料集約・LLM 整形の手前でハード静音／夜間静音を安価に先読みし、抑制中の
  毎 tick 反復（最長 6h の失効窓 × 毎分）を止める。**配達可否の正式なゲート判定は従来どおり
  `deliver_event` 内の 1 回**で、二重ゲートにはしない

詳細は docs/architecture.md v1.6 / docs/regular-talk-design.md v1.1 を参照。

## 既知の制限 (このリリースでは対応しない)

v0.2.0 の「既知の制限」をそのまま引き継ぐ。加えて本リリース由来:

1. **地名検索の日本語表記ゆれ**: Geocoding（Open-Meteo / GeoNames）は表記によって 0 件に
   なることがある（「東京」→ 0 件・「東京都」→ OK、「渋谷」→ OK・「渋谷区」→ 0 件 など）。
   候補を都道府県付きで複数表示し、検索ヒントと 0 件時の言い換え案内で補っている
2. **advanced の言い回し整形はコスト上限の閾値チェックを経由しない**（80% 警告・自動降格は
   チャット経路でのみ発火する）。「1 日最大 2 回・短い言い換え」という発火頻度を前提に受容した
   （設計 §13.1-7）。抑制中の毎 tick 反復は上記の静音プリフィルタで防いでいる
3. 天気は**単一地点のみ**。複数地点・週間予報の読み上げ・気象警報／注意報・気温差アラートは
   将来課題（spec §6.6）
4. 定例会話の配達判定は 60 秒間隔のポーリングのため、配達タイミングが数十秒ぶれることがある

## 動作要件

v0.2.0 と同一。天気・カレンダー（URL）・advanced 会話はインターネット接続を使う（いずれも
既定オフ、またはユーザーが設定したときのみ）。

- Windows 10/11 x64
- 一般機能: 任意の x86_64 CPU、3 GB 以上の空き RAM
- voicevox_core (推奨): 数百 MB のディスク + 一般的な CPU
- Irodori-TTS (任意): NVIDIA GPU (CUDA 12.8 対応、VRAM 4 GB 以上)、10 GB 以上のディスク

## アップグレード手順

v0.2.x からのアップグレードは、新しいインストーラを上書き実行するだけでよい（currentUser
モード）。`%APPDATA%\ugg\` の既存データは引き継がれる。**DB マイグレーションは発生しない**
（スキーマは v8 のまま）。新機能（定例会話・天気・降雨の一言）はすべて既定オフのオプトイン
なので、アップグレードで挙動が勝手に増えることはない。

v0.0.3 プロトタイプからの DB マイグレーションは提供しない。
