# AI Model Routing / Token ROI 詳細

目的: メインセッションが Fable5 でも、全作業を Fable5 で実行せず、出力品質を下げずにトークン消費を抑える。
Fable5 は「全部を実行するモデル」ではなく、設計・判断・監査・レビュー・最終品質担保に集中させる。

## 役割分担

| 作業 | 担当 |
|---|---|
| アーキテクチャ設計・スキーマ定義・API 契約 | Fable5 |
| UX・コスト・セキュリティに関わる判断 | Fable5 |
| サブエージェント成果物のレビュー・統合・最終品質確認 | Fable5 |
| 仕様が固まったモジュールの実装 | Opus/Sonnet サブエージェント |
| 既存コードの調査・読解・整理 | Opus/Sonnet サブエージェント |
| テスト追加・差分確認・軽微な修正・機械的リファクタ | Opus/Sonnet サブエージェント |

## 運用ルール
1. サブエージェントに渡すのは「対象ファイル・契約（docs/architecture.md の該当節）・完了条件」のみ。会話履歴全体は渡さない。
2. サブエージェントの報告フォーマットは固定（docs/subagent_prompt_templates.md 参照）:
   **変更内容 / 判断理由 / 懸念点 / Fable5 で判断すべきこと**
3. サブエージェントが設計判断に踏み込みそうな場合は、判断を保留して「Fable5 で判断すべきこと」として返させる。
4. 実装完了後、Fable5 が `cargo check` / `tsc --noEmit` と契約整合性（コマンド名・型・イベント名）を必ず検証する。
5. 節約の対象は中間作業のみ。最終成果物の品質を下げる節約はしない。
